Story

2.1 — Fix Repository Clean Architecture Violation

Remove the data-layer dependency from the domain layer by changing BikeInfoRepository to return Result<BikeInfo> instead of Result<BikeInfoSnapshotDto>, relocating the mapper from domain/mapper/ to data/mapper/, and simplifying the use case to a pass-through.

Change Proposed

Source: opsx:propose

Exploration Findings

Investigated all 6 source files and 3 test files affected by the architecture violation. Confirmed the dependency flow:

  • BikeInfoRepository (domain) imports BikeInfoSnapshotDto (data) — violates dependency rule
  • BikeInfoMapper lives in domain/mapper/ despite converting data-layer types
  • GetBikeInfoUseCase performs DTO-to-domain mapping that should be the repository's responsibility
  • LocalBikeInfoRepository.runCatching block already wraps JSON parsing — adding .toDomain() inside it will catch mapping errors too
  • The use case's public API (suspend operator fun invoke(): Result<BikeInfo>) remains unchanged — no downstream impact

Questions and Answers

No ambiguities found — the user story is fully specified with detailed current-state vs target-state descriptions, file paths, and acceptance criteria. Proceeded directly to proposal.

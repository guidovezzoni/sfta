# 2.1 Fix Repository Clean Architecture Violation

## User Story

**As a** developer maintaining the rider dashboard,
**I want** the domain layer to have no dependency on data-layer types,
**So that** the architecture follows Clean Architecture principles and the layers can evolve independently.

## Description

The `BikeInfoRepository` interface in the domain layer currently imports and returns `Result<BikeInfoSnapshotDto>` — a data-layer DTO. In Clean Architecture, the domain layer must not depend on the data layer. The interface should return `Result<BikeInfo>` (the domain model), with the DTO-to-domain mapping happening inside the repository implementation.

This also means the mapper, which converts data-layer types to domain types, belongs in the data layer rather than the domain layer.

## Scope

### Repository Interface

- **`domain/repository/BikeInfoRepository.kt`**: change return type from `Result<BikeInfoSnapshotDto>` to `Result<BikeInfo>`. Remove the import of `BikeInfoSnapshotDto`.

### Repository Implementation

- **`data/repository/LocalBikeInfoRepository.kt`**: after parsing the JSON into `BikeInfoSnapshotDto`, apply the DTO-to-domain mapping before returning. The method now returns `Result<BikeInfo>`.

### Mapper Relocation

- **Move `domain/mapper/BikeInfoMapper.kt`** to **`data/mapper/BikeInfoMapper.kt`** using `git mv`. The mapper converts data-layer types (DTOs) to domain-layer types, so it belongs in the data layer. The extension function `BikeInfoSnapshotDto.toDomain()` and its logic remain unchanged.

### Use Case Simplification

- **`domain/usecase/GetBikeInfoUseCase.kt`**: becomes a pass-through — calls `repository.getBikeInfoSnapshot()` and returns the result directly (no `.map { it.toDomain() }`). Kept as a thin wrapper to maintain the use case pattern for future orchestration.

### Spec and Documentation

- **`openspec/specs/bike-telemetry-data/spec.md`**: update the Repository requirement to return `Result<BikeInfo>`, update the Use Case requirement to reflect pass-through behaviour, update the Mapper requirement to note its location in the data layer.
- **User story 1.1**: update the Repository interface, Mapper, and Use Case sections to reflect the corrected architecture.

### Tests

- **`data/repository/LocalBikeInfoRepositoryTest.kt`**: update success-path assertions to verify domain model fields (`BikeInfo`) instead of DTO fields. Failure-path tests remain structurally the same.
- **Move `domain/mapper/BikeInfoMapperTest.kt`** to **`data/mapper/BikeInfoMapperTest.kt`** using `git mv`. Tests remain unchanged, only the package moves.
- **`domain/usecase/GetBikeInfoUseCaseTest.kt`**: simplify — the mock `BikeInfoRepository` now returns `Result<BikeInfo>` directly. Remove DTO construction from tests. Verify success delegates to repository; verify failure is propagated.

## Acceptance Criteria (BDD)

### Scenario: Repository interface has no data-layer imports
```
GIVEN the domain/repository/BikeInfoRepository.kt file
WHEN its imports are inspected
THEN it contains no imports from the data package
```

### Scenario: Repository returns domain model on success
```
GIVEN a valid bike_info_snapshot.json is available via AssetManager
WHEN getBikeInfoSnapshot() is called
THEN it returns Result.success containing a BikeInfo domain model with all fields correctly mapped
```

### Scenario: Repository handles missing file gracefully
```
GIVEN the asset file does not exist
WHEN getBikeInfoSnapshot() is called
THEN it returns Result.failure with an appropriate exception
```

### Scenario: Repository handles malformed JSON gracefully
```
GIVEN the asset file contains invalid JSON
WHEN getBikeInfoSnapshot() is called
THEN it returns Result.failure with a serialisation exception
```

### Scenario: Mapper handles unknown enum values with UNKNOWN fallback
```
GIVEN a DTO with unrecognised enum string values
WHEN the repository maps it to a domain model
THEN the corresponding enum fields fall back to UNKNOWN
```

### Scenario: Use case delegates to repository
```
GIVEN the repository returns a successful Result<BikeInfo>
WHEN GetBikeInfoUseCase is invoked
THEN it returns the same Result unchanged
```

### Scenario: Use case propagates failure from repository
```
GIVEN the repository returns a failure Result
WHEN GetBikeInfoUseCase is invoked
THEN it returns the same Result.failure
```

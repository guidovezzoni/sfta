# 2.2 Domain Model Refinements

## User Story

**As a** developer building the rider dashboard,
**I want** the domain model to use a proper timestamp type and include fault codes,
**So that** the UI can format times locale-appropriately and the app exposes all available diagnostic data.

## Description

Two domain model improvements bundled into a single story:

1. **Timestamp type (C1)**: `BikeInfo.timestamp` is currently a raw `String`. For a production app, this should be a `kotlinx.datetime.Instant` so the UI can format it locale-appropriately and the domain can reason about time.
2. **Fault codes (C2)**: `DiagnosticsDto.faultCodes` is parsed from JSON but silently dropped in the mapper — `DiagnosticsInfo` only contains `warnings`. The field should be mapped to the domain model so the data is available for future use.

**Dependency:** This story assumes story 2.1 (Clean Architecture fix) is complete, so the mapper is in `data/mapper/`.

## Scope

### C1: Parse Timestamp to Instant

#### Dependency

- **`gradle/libs.versions.toml`**: add `kotlinxDatetime` version (`0.6.2`) and library entry (`org.jetbrains.kotlinx:kotlinx-datetime`).
- **`app/build.gradle.kts`**: add `implementation(libs.kotlinx.datetime)`.

#### Domain Model

- **`domain/model/BikeInfo.kt`**: change `timestamp: String` to `timestamp: Instant` (from `kotlinx.datetime`).

#### Mapper

- **`data/mapper/BikeInfoMapper.kt`**: parse the ISO-8601 timestamp string to `Instant` using `Instant.parse()`. If parsing fails, the `runCatching` in the repository will propagate the failure.

### C2: Add Fault Codes to Domain Model

#### Domain Model

- **`domain/model/DiagnosticsInfo.kt`**: add `faultCodes: List<String>` field.

#### Mapper

- **`data/mapper/BikeInfoMapper.kt`**: map `diagnosticsDto.faultCodes` to `diagnosticsInfo.faultCodes` (direct pass-through).

### Spec and Documentation

- **`openspec/specs/bike-telemetry-data/spec.md`**: update the Domain Models requirement to reflect `timestamp: Instant` and `DiagnosticsInfo.faultCodes`. Update the Mapper requirement to include timestamp parsing and `faultCodes` mapping. Remove the note about intentionally excluding `faultCodes`.
- **User story 1.1**: update the Domain Models and Mapper sections accordingly.

### Tests

- **`data/mapper/BikeInfoMapperTest.kt`**: update timestamp assertion to compare `Instant` values. Add assertion for `faultCodes` mapping. Add test for empty `faultCodes` list.
- **`data/repository/LocalBikeInfoRepositoryTest.kt`**: update timestamp assertion to compare `Instant` values.

## Acceptance Criteria (BDD)

### Scenario: Timestamp is parsed to Instant
```
GIVEN a DTO with timestamp "2025-05-19T10:32:45Z"
WHEN the mapper converts it to a domain model
THEN BikeInfo.timestamp is an Instant representing 2025-05-19T10:32:45Z
```

### Scenario: Fault codes are mapped to domain model
```
GIVEN a DTO with faultCodes ["F001", "F002"]
WHEN the mapper converts it to a domain model
THEN DiagnosticsInfo.faultCodes contains ["F001", "F002"]
```

### Scenario: Empty fault codes list is preserved
```
GIVEN a DTO with an empty faultCodes list
WHEN the mapper converts it to a domain model
THEN DiagnosticsInfo.faultCodes is an empty list
```

### Scenario: Existing mapper behaviour is preserved
```
GIVEN a fully populated BikeInfoSnapshotDto
WHEN the mapper converts it to a domain model
THEN all fields including the new timestamp type and faultCodes are correctly mapped
```

## Impact on Story 1.2

- Story 1.2's ViewModel will receive `Instant` for the timestamp and must format it for display (e.g., using `kotlinx-datetime` formatters or converting to `java.time` for locale-aware formatting).
- Story 1.2's `DashboardUiState` may include a pre-formatted timestamp string, with formatting done in the ViewModel.
- Fault codes could be displayed in a future diagnostics screen but are not required in story 1.2's dashboard layout.

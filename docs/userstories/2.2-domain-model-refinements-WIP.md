# 2.2 Domain Model Refinements

## User Story

**As a** developer building the rider dashboard,
**I want** the domain model to use a proper timestamp type and include fault codes,
**So that** the UI can format times locale-appropriately and the app exposes all available diagnostic data.

## Description

Two domain model improvements bundled into a single story:

1. **Timestamp type (C1)**: `BikeInfo.timestamp` is currently a raw `String`. For a production app, this should be a `kotlinx.datetime.Instant` so the UI can format it locale-appropriately and the domain can reason about time.
2. **Fault codes (C2)**: `DiagnosticsDto.faultCodes` is parsed from JSON but silently dropped in the mapper — `DiagnosticsInfo` only contains `warnings`. The field should be mapped to the domain model so the data is available for future use.

**Dependency:** This story assumes story 2.1 (Clean Architecture fix) is complete, so the mapper is in `data/mapper/`.

## Scope

### C1: Parse Timestamp to Instant

#### Dependency

- **`gradle/libs.versions.toml`**: add `kotlinxDatetime` version (`0.6.2`) and library entry (`org.jetbrains.kotlinx:kotlinx-datetime`).
- **`app/build.gradle.kts`**: add `implementation(libs.kotlinx.datetime)`.

#### Domain Model

- **`domain/model/BikeInfo.kt`**: change `timestamp: String` to `timestamp: Instant` (from `kotlinx.datetime`). Add `import kotlinx.datetime.Instant`.

#### Mapper

- **`data/mapper/BikeInfoMapper.kt`**: parse the ISO-8601 timestamp string to `Instant` using `Instant.parse(timestamp)`. If parsing fails, the `runCatching` in the repository will propagate the failure. Add `import kotlinx.datetime.Instant`.

### C2: Add Fault Codes to Domain Model

#### Domain Model

- **`domain/model/DiagnosticsInfo.kt`**: add `faultCodes: List<String>` field before the existing `warnings` field.

#### Mapper

- **`data/mapper/BikeInfoMapper.kt`**: map `diagnostics.faultCodes` to `diagnosticsInfo.faultCodes` (direct pass-through). The updated mapper call becomes:
  ```kotlin
  diagnostics = DiagnosticsInfo(
      faultCodes = diagnostics.faultCodes,
      warnings = diagnostics.warnings.map { ... },
  )
  ```

### Spec and Documentation

- **`openspec/specs/bike-telemetry-data/spec.md`**: update the Domain Models requirement to reflect `timestamp: Instant` and `DiagnosticsInfo.faultCodes`. Update the Mapper requirement to include timestamp parsing and `faultCodes` mapping. Remove the note about intentionally excluding `faultCodes`.
- **User story 1.1**: update the Domain Models and Mapper sections accordingly.

### Tests

- **`data/mapper/BikeInfoMapperTest.kt`**:
  - Update timestamp assertion in the fully-populated test: compare against `Instant.parse("2025-05-19T10:32:45Z")` instead of the raw string.
  - Add assertion for `faultCodes` mapping (e.g. `assertEquals(listOf("F001"), result.diagnostics.faultCodes)`).
  - Update the empty warnings test to also assert `faultCodes` is empty.
  - Add a dedicated test for empty `faultCodes` list.
- **`data/repository/LocalBikeInfoRepositoryTest.kt`**: update timestamp assertion to compare `Instant` values. Add `import kotlinx.datetime.Instant`.

## Endpoints

No network endpoints required. Data is loaded from a local JSON asset (`assets/bike_info_snapshot.json`).

| `domain/model/DiagnosticsInfo.kt` | `faultCodes` | *(missing)* | `List<String>` |
| `data/mapper/BikeInfoMapper.kt` | `timestamp` mapping | `timestamp = timestamp` (pass-through) | `timestamp = Instant.parse(timestamp)` |

## Acceptance Criteria (BDD)

### Scenario: Timestamp is parsed to Instant
```
GIVEN a DTO with timestamp "2025-05-19T10:32:45Z"
WHEN the mapper converts it to a domain model
THEN BikeInfo.timestamp is an Instant representing 2025-05-19T10:32:45Z
```

### Scenario: Fault codes are mapped to domain model
```
GIVEN a DTO with faultCodes ["F001", "F002"]
WHEN the mapper converts it to a domain model
THEN DiagnosticsInfo.faultCodes contains ["F001", "F002"]
```

### Scenario: Empty fault codes list is preserved
```
GIVEN a DTO with an empty faultCodes list
WHEN the mapper converts it to a domain model
THEN DiagnosticsInfo.faultCodes is an empty list
```

### Scenario: Existing mapper behaviour is preserved
```
GIVEN a fully populated BikeInfoSnapshotDto
WHEN the mapper converts it to a domain model
THEN all fields including the new timestamp type and faultCodes are correctly mapped
```

## Security Considerations

- **No new attack surface**: This story adds a timestamp parsing library (`kotlinx-datetime`) which operates on in-memory strings only — no network, file, or user input vectors are introduced.
- **No PII impact**: Timestamps are machine-generated telemetry data, not personally identifiable.
- **Fault codes exposure**: Fault codes (`F001`, etc.) are diagnostic identifiers, not sensitive. No additional access controls are needed.

## Performance Considerations

- **`Instant.parse()` overhead**: Negligible — parsing a single ISO-8601 string is O(1) and sub-microsecond. Called once per snapshot load.
- **No additional allocations**: The `faultCodes` list is a direct reference pass-through from the DTO, not a copy.

## GDPR and Sensitive Information

- Fault codes and timestamps are machine telemetry, not personal data. No GDPR implications.
- No new data is collected, stored, or transmitted.

## Non-Functional Requirements

- **Backward compatibility**: The JSON asset format is unchanged. Only the domain-side representation changes.
- **Error handling**: Invalid timestamp strings will cause `Instant.parse()` to throw `IllegalArgumentException`, which is caught by the repository's `runCatching`. No new error paths are exposed to the UI.
- **Library size**: `kotlinx-datetime` is a lightweight library (~100 KB) with no transitive dependencies beyond the Kotlin stdlib.

## Impact on Story 1.2

- Story 1.2's ViewModel will receive `Instant` for the timestamp and must format it for display (e.g., using `kotlinx-datetime` formatters or converting to `java.time` for locale-aware formatting).
- Story 1.2's `DashboardUiState` may include a pre-formatted timestamp string, with formatting done in the ViewModel.
- Fault codes could be displayed in a future diagnostics screen but are not required in story 1.2's dashboard layout.

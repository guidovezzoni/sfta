# 2.3 Introduce Hilt Dependency Injection

## User Story

**As a** developer scaling the rider dashboard app,
**I want** Hilt managing the dependency graph,
**So that** new features can declare their dependencies declaratively without manual wiring.

## Description

Manual DI (constructing the dependency graph in `MainActivity`) works at the current project size but won't scale. The Android guidelines already say "introduce Hilt when the project grows." With stories 2.1 and 2.2 complete, the data and domain layers are stable — this is the right time to introduce Hilt before building the Dashboard UI (story 1.2).

**Dependency:** This story assumes stories 2.1 and 2.2 are complete.

## Scope

### Gradle Configuration

- **`gradle/libs.versions.toml`**: add Hilt version, `hilt-android` library, `hilt-compiler` library (KSP annotation processor), `hilt-android-gradle-plugin` plugin, and KSP plugin (version matching Kotlin).
- **`build.gradle.kts` (project)**: add `hilt-android-gradle-plugin` and `ksp` plugin with `apply false`.
- **`app/build.gradle.kts`**: apply `hilt-android` and `ksp` plugins. Add `implementation(libs.hilt.android)` and `ksp(libs.hilt.compiler)`.

### Application Class

- **Create `SftaApplication.kt`**: annotate with `@HiltAndroidApp`.
- **`AndroidManifest.xml`**: add `android:name=".SftaApplication"` to the `<application>` element.

### DI Module

- **Create `di/AppModule.kt`**: Hilt `@Module` + `@InstallIn(SingletonComponent::class)` providing:
  - `AssetManager` (from application context)
  - `BikeInfoRepository` bound to `LocalBikeInfoRepository` (`@Singleton`)
  - `GetBikeInfoUseCase`

### Existing Classes

- **`data/repository/LocalBikeInfoRepository.kt`**: add `@Inject constructor`.
- **`domain/usecase/GetBikeInfoUseCase.kt`**: add `@Inject constructor`.

### Activity

- **`MainActivity.kt`**: annotate with `@AndroidEntryPoint`. Remove any manual dependency construction.

### Spec and Documentation

- **`openspec/specs/bike-telemetry-data/spec.md`**: update the Repository and Use Case requirements to note Hilt-provided dependencies.
- **User story 1.1**: update the relevant sections to reflect Hilt injection.

## Acceptance Criteria (BDD)

### Scenario: Project compiles with Hilt dependencies
```
GIVEN the Hilt plugin and dependencies have been added
WHEN ./gradlew assembleDebug is executed
THEN the project compiles without errors
```

### Scenario: Application class is configured for Hilt
```
GIVEN SftaApplication is annotated with @HiltAndroidApp
AND AndroidManifest.xml references SftaApplication
WHEN the app is launched
THEN Hilt initialises the dependency graph without errors
```

### Scenario: Repository is provided by Hilt
```
GIVEN AppModule provides BikeInfoRepository as a singleton
WHEN a class requests BikeInfoRepository via @Inject
THEN Hilt provides a LocalBikeInfoRepository instance
```

### Scenario: Use case is injectable
```
GIVEN GetBikeInfoUseCase has an @Inject constructor
WHEN a ViewModel requests GetBikeInfoUseCase
THEN Hilt provides an instance with the repository injected
```

### Scenario: Existing tests continue to pass
```
GIVEN all existing unit tests for repository, mapper, and use case
WHEN ./gradlew test is executed
THEN all tests pass without modification
```

## Test Plan

### Unit Tests

Existing tests for `LocalBikeInfoRepository`, `BikeInfoMapper`, and `GetBikeInfoUseCase` should pass unchanged — they construct dependencies manually in test setup, independent of Hilt.

### Build Verification

- `./gradlew assembleDebug` compiles with Hilt annotation processing.
- `./gradlew test` passes all existing tests.
- `./gradlew check` passes code quality checks.

- Story 1.2 no longer needs `DashboardViewModelFactory` — the ViewModel should use `@HiltViewModel` with `@Inject constructor`.
- Story 1.2's Phase 7 (Wiring) simplifies: `MainActivity` uses `hiltViewModel()` instead of manual factory construction.

# 1.1 Data Infrastructure

## User Story

**As a** developer setting up the rider dashboard,
**I want** the project infrastructure, data layer, and domain layer in place,
**So that** the app can load, parse, and map the bike telemetry snapshot ready for the UI to consume.

## Description

This story establishes the foundational layers of the app: Gradle dependencies for serialisation and testing, the JSON asset containing bike telemetry data, the data layer (DTOs + repository), and the domain layer (domain models, mapper, use case). The app is also locked to landscape orientation.

The JSON snapshot (`bike_info_snapshot.json`) is a static asset bundled with the app. It represents a single point-in-time telemetry reading from a Stark VARG electric motorbike, including battery state, motor metrics, ride settings, session stats, and diagnostics/warnings.

### Phase 2: Data Layer

#### DTOs -- `app/src/main/java/com/guidovezzoni/sfta/data/model/`

One class per file. Each class is `@Serializable` and uses `@SerialName` for snake_case JSON keys.

| DTO class | Fields |
|-----------|--------|
| `BikeInfoSnapshotDto` | `bike: BikeDto`, `timestamp: String`, `battery: BatteryDto`, `motor: MotorDto`, `rideSettings: RideSettingsDto`, `session: SessionDto`, `diagnostics: DiagnosticsDto` |
| `BikeDto` | `model: String`, `variant: String`, `firmwareVersion: String`, `imageUrl: String` |
| `BatteryDto` | `stateOfChargePct: Int`, `estimatedRangeKm: Int`, `temperatureC: Double`, `chargingState: String` |
| `MotorDto` | `powerHp: Double`, `temperatureC: Double`, `currentSpeedKmh: Double` |
| `RideSettingsDto` | `powerMap: String`, `maxPowerHp: Int`, `engineBrakingPct: Int`, `regenPct: Int` |
| `SessionDto` | `durationS: Long`, `distanceKm: Double`, `maxSpeedKmh: Double` |
| `DiagnosticsDto` | `faultCodes: List<String>`, `warnings: List<WarningDto>` |
| `WarningDto` | `code: String`, `message: String`, `severity: String` |

#### Repository interface -- `app/src/main/java/com/guidovezzoni/sfta/domain/repository/BikeInfoRepository.kt`

```kotlin
interface BikeInfoRepository {
    suspend fun getBikeInfoSnapshot(): Result<BikeInfoSnapshotDto>
}
```

#### Repository implementation -- `app/src/main/java/com/guidovezzoni/sfta/data/repository/LocalBikeInfoRepository.kt`

- Constructor receives `AssetManager`.
- Uses `Json { ignoreUnknownKeys = true }` for forward compatibility.
- Reads the asset file as a string, then deserialises with `kotlinx.serialization`.
- Wraps the entire operation in `runCatching`.
- Runs on `Dispatchers.IO` (file I/O must not block the main thread).

> **Updated in story 2.3:** `LocalBikeInfoRepository` is now constructed and provided as a `@Singleton` via `AppModule` (`@Provides`). The `AssetManager` is supplied by Hilt from the application context.

### Phase 3: Domain Layer

#### Domain models -- `app/src/main/java/com/guidovezzoni/sfta/domain/model/`

One class per file:

| Class | Fields |
|-------|--------|
| `BikeInfo` | `bike: BikeDetails`, `timestamp: Instant`, `battery: BatteryInfo`, `motor: MotorInfo`, `rideSettings: RideSettingsInfo`, `session: SessionInfo`, `diagnostics: DiagnosticsInfo` |
| `BikeDetails` | `model: String`, `variant: String`, `firmwareVersion: String`, `imageUrl: String` |
| `DiagnosticsInfo` | `faultCodes: List<String>`, `warnings: List<WarningInfo>` |
| `BatteryInfo` | `stateOfChargePercent: Int`, `estimatedRangeKm: Int`, `temperatureCelsius: Double`, `chargingState: ChargingState` |
| `MotorInfo` | `powerHp: Double`, `temperatureCelsius: Double`, `currentSpeedKmh: Double` |
| `RideSettingsInfo` | `powerMap: PowerMap`, `maxPowerHp: Int`, `engineBrakingPercent: Int`, `regenPercent: Int` |
| `SessionInfo` | `durationSeconds: Long`, `distanceKm: Double`, `maxSpeedKmh: Double` |
| `WarningInfo` | `code: String`, `message: String`, `severity: WarningSeverity` |

Enums (each with an `UNKNOWN` fallback):

| Enum | Values |
|------|--------|
| `ChargingState` | `DISCHARGING`, `UNKNOWN` |
| `PowerMap` | `ENDURO`, `UNKNOWN` |
| `WarningSeverity` | `WARNING`, `UNKNOWN` |

> **Note:** These enums only list values confirmed in the JSON snapshot. Full API documentation was not available at the time of writing, so additional values may exist.

#### Mapper -- `app/src/main/java/com/guidovezzoni/sfta/domain/mapper/BikeInfoMapper.kt`

Extension function `BikeInfoSnapshotDto.toDomain(): BikeInfo`.

- Maps each DTO field to its domain counterpart, preserving the nested structure.
- `BikeDto` maps to `BikeDetails`; `DiagnosticsDto` maps to `DiagnosticsInfo` including `faultCodes` (direct pass-through) and `warnings` (with enum conversion).
- Parses `BikeInfoSnapshotDto.timestamp` from ISO-8601 string to `kotlinx.datetime.Instant` using `Instant.parse()`.
- Enum conversion uses a case-insensitive lookup with `UNKNOWN` fallback (e.g. `ChargingState.entries.find { it.name.equals(dto.chargingState, ignoreCase = true) } ?: ChargingState.UNKNOWN`).
- Empty `warnings` list maps to `emptyList()`.

#### Use case -- `app/src/main/java/com/guidovezzoni/sfta/domain/usecase/GetBikeInfoUseCase.kt`

```kotlin
class GetBikeInfoUseCase(private val repository: BikeInfoRepository) {
    suspend operator fun invoke(): Result<BikeInfo> {
        return repository.getBikeInfoSnapshot().map { it.toDomain() }
    }
}
```

> **Updated in story 2.3:** `GetBikeInfoUseCase` now declares `@Inject constructor` so Hilt can auto-resolve it from the module-provided `BikeInfoRepository`.

## Endpoints

No network endpoints required. Data is loaded from a local JSON asset (`assets/bike_info_snapshot.json`).

## Security Considerations

- **External URL in data**: The `image_url` field in the JSON contains a remote URL (`https://assets.starkfuture.com/...`). This story does not load or display it. When the UI story (1.2) uses it, it must be loaded over HTTPS only and should not be used to construct any other requests.
- **No PII**: The telemetry snapshot contains no personally identifiable information.
- **No network calls**: The repository reads from a bundled asset -- no server-side attack surface.

## Performance Considerations

- **IO dispatcher**: JSON parsing runs on `Dispatchers.IO` to avoid blocking the main thread.
- **Single read**: The snapshot is small (~1 KB) -- no caching layer is needed at this stage.
- **`ignoreUnknownKeys`**: Enables forward compatibility if the JSON gains new fields without requiring an app update.

## GDPR and Sensitive Information

- The telemetry snapshot contains no personal data, no device identifiers, and no location data.
- No data is transmitted over the network. No data is persisted to local storage beyond the bundled read-only asset.
- No GDPR concerns apply at this stage.

## Non-Functional Requirements

- **Minimum SDK**: 24 (as per current `build.gradle.kts`).
- **Orientation lock**: `sensorLandscape` allows both landscape orientations (left and right) while preventing portrait. This is appropriate for a dashboard-style display.
- **Error handling**: All operations use Kotlin `Result` type. No exceptions propagate uncaught.
- **Testability**: Repository takes `AssetManager` as a constructor parameter for easy mocking in tests. Since story 2.3, Hilt also supports test-specific modules (`hilt-android-testing`) for dependency substitution in integration tests.

## Test Plan

### Unit Tests (`app/src/test/java/com/guidovezzoni/sfta/`)

| Test File | Location | Covers |
|-----------|----------|--------|
| `data/repository/LocalBikeInfoRepositoryTest.kt` | `app/src/test/` | Valid JSON returns success; missing file returns failure; malformed JSON returns failure |
| `domain/mapper/BikeInfoMapperTest.kt` | `app/src/test/` | Complete DTO-to-domain mapping; unknown enum fallback to `UNKNOWN`; empty warnings list |
| `domain/usecase/GetBikeInfoUseCaseTest.kt` | `app/src/test/` | Success path returns mapped `BikeInfo`; failure is propagated unchanged |

### Testing approach

- **Repository tests**: Use `mockk` to mock `AssetManager`. Return a `ByteArrayInputStream` with valid/invalid/missing JSON. Use `runTest` with `UnconfinedTestDispatcher`.
- **Mapper tests**: Create `BikeInfoSnapshotDto` instances directly (no mocking needed). Assert each field is mapped correctly. Test enum fallback with unrecognised string values.
- **Use case tests**: Use `mockk` to mock `BikeInfoRepository`. Verify success maps through the mapper. Verify failure is returned as-is.

# 1.2 Data Model Null Safety

## User Story

**As a** developer integrating real-time telemetry from the Stark VARG,
**I want** every data model field to handle null values gracefully,
**So that** the app never crashes when the bike reports incomplete telemetry during real data collection.

## Background

The JSON snapshot no longer provides `current_speed_kmh` in the motor object — this field must be removed from the DTO, domain model, and mapper.

Additionally, when real data collection is implemented, any JSON field could be `null` (sensor offline, data not yet available, partial telemetry). The data layer must deserialise these without crashing, and the domain layer must surface nullability so the UI can decide how to present missing values.

## Scope

### Phase 1: Remove `currentSpeedKmh`

- **`data/model/MotorDto.kt`**: remove `currentSpeedKmh` field
- **`domain/model/MotorInfo.kt`**: remove `currentSpeedKmh` field
- **`data/mapper/BikeInfoMapper.kt`**: remove `currentSpeedKmh` mapping line

### Phase 2: Make DTO fields nullable

All DTO fields become nullable with `= null` defaults. This tells kotlinx-serialization the field is optional — missing or `null` in JSON both produce `null` in the DTO.

- **`data/model/BikeInfoSnapshotDto.kt`**: all top-level sub-objects nullable (`bike: BikeDto? = null`, `timestamp: String? = null`, etc.)
- **`data/model/BikeDto.kt`**: all fields nullable (`model: String? = null`, `variant: String? = null`, `firmwareVersion: String? = null`, `imageUrl: String? = null`)
- **`data/model/BatteryDto.kt`**: all fields nullable (`stateOfChargePct: Int? = null`, `estimatedRangeKm: Int? = null`, `temperatureC: Double? = null`, `chargingState: String? = null`)
- **`data/model/MotorDto.kt`**: all remaining fields nullable (`powerHp: Double? = null`, `temperatureC: Double? = null`)
- **`data/model/RideSettingsDto.kt`**: all fields nullable (`powerMap: String? = null`, `maxPowerHp: Int? = null`, `engineBrakingPct: Int? = null`, `regenPct: Int? = null`)
- **`data/model/SessionDto.kt`**: all fields nullable (`durationS: Long? = null`, `distanceKm: Double? = null`, `maxSpeedKmh: Double? = null`)
- **`data/model/DiagnosticsDto.kt`**: all fields nullable (`faultCodes: List<String>? = null`, `warnings: List<WarningDto>? = null`)
- **`data/model/WarningDto.kt`**: all fields nullable (`code: String? = null`, `message: String? = null`, `severity: String? = null`)

### Phase 3: Make domain model fields nullable

Domain models mirror the DTO nullability so the UI layer can decide how to render missing data (e.g. "--", hide a panel, show a placeholder).

- **`domain/model/BikeInfo.kt`**: all fields nullable (`bike: BikeDetails?`, `timestamp: Instant?`, `battery: BatteryInfo?`, `motor: MotorInfo?`, `rideSettings: RideSettingsInfo?`, `session: SessionInfo?`, `diagnostics: DiagnosticsInfo?`)
- **`domain/model/BikeDetails.kt`**: all fields nullable (`model: String?`, `variant: String?`, `firmwareVersion: String?`, `imageUrl: String?`)
- **`domain/model/BatteryInfo.kt`**: all fields nullable (`stateOfChargePercent: Int?`, `estimatedRangeKm: Int?`, `temperatureCelsius: Double?`, `chargingState: ChargingState?`)
- **`domain/model/MotorInfo.kt`**: all fields nullable (`powerHp: Double?`, `temperatureCelsius: Double?`)
- **`domain/model/RideSettingsInfo.kt`**: all fields nullable (`powerMap: PowerMap?`, `maxPowerHp: Int?`, `engineBrakingPercent: Int?`, `regenPercent: Int?`)
- **`domain/model/SessionInfo.kt`**: all fields nullable (`durationSeconds: Long?`, `distanceKm: Double?`, `maxSpeedKmh: Double?`)
- **`domain/model/DiagnosticsInfo.kt`**: all fields nullable (`faultCodes: List<String>?`, `warnings: List<WarningInfo>?`)
- **`domain/model/WarningInfo.kt`**: all fields nullable (`code: String?`, `message: String?`, `severity: WarningSeverity?`)

Note: enum fields (`ChargingState?`, `PowerMap?`, `WarningSeverity?`) are nullable. `null` means "no data from sensor"; `UNKNOWN` means "data received but value not recognised". This is a meaningful distinction.

### Phase 4: Update mapper

- **`data/mapper/BikeInfoMapper.kt`**: update `toDomain()` to propagate nulls:
  - Null sub-objects in DTO → null in domain (e.g. `battery = dto.battery?.let { ... }`)
  - Null primitive fields → null in domain (direct assignment)
  - Null enum strings → null (not `UNKNOWN`; `UNKNOWN` is reserved for non-null unrecognised values)
  - Null timestamp string → null `Instant`
  - Null lists → null (not empty list)

### Phase 5: Update tests

- **`data/mapper/BikeInfoMapperTest.kt`**:
  - Remove `currentSpeedKmh` assertion from existing full-mapping test
  - Add test: JSON with all-null leaf fields → domain model has all-null fields
  - Add test: JSON with null sub-objects (e.g. no `battery`) → domain `battery` is null
  - Add test: null enum string → domain enum field is null (distinct from `UNKNOWN`)
  - Add test: null timestamp → domain `timestamp` is null
  - Add test: null lists → domain list fields are null
- **`data/repository/LocalBikeInfoRepositoryTest.kt`**: verify existing tests still pass with the updated models
- **`domain/usecase/GetBikeInfoUseCaseTest.kt`**: verify existing tests still pass

## Acceptance Criteria (BDD)

### Scenario: Fully populated JSON parses correctly (regression)
```
GIVEN a JSON snapshot with all fields populated
WHEN the snapshot is parsed and mapped to domain
THEN all domain model fields contain the expected non-null values
```

### Scenario: JSON with null leaf fields does not crash
```
GIVEN a JSON snapshot where individual fields are null (e.g. battery temperature, motor power)
WHEN the snapshot is parsed and mapped to domain
THEN the corresponding domain fields are null
AND no exception is thrown
```

### Scenario: JSON with missing sub-objects does not crash
```
GIVEN a JSON snapshot where an entire sub-object is null (e.g. "session": null)
WHEN the snapshot is parsed and mapped to domain
THEN the corresponding domain field is null (e.g. session = null)
AND no exception is thrown
```

### Scenario: Null enum string maps to null, not UNKNOWN
```
GIVEN a JSON snapshot where an enum field is null (e.g. "charging_state": null)
WHEN the snapshot is parsed and mapped to domain
THEN the domain enum field is null
AND UNKNOWN is not used (UNKNOWN is reserved for non-null unrecognised values)
```

### Scenario: Unrecognised enum string still maps to UNKNOWN
```
GIVEN a JSON snapshot where an enum field has an unrecognised value (e.g. "charging_state": "supercharging")
WHEN the snapshot is parsed and mapped to domain
THEN the domain enum field is UNKNOWN (existing behaviour preserved)
```

### Scenario: current_speed_kmh is removed from motor
```
GIVEN the motor object in the JSON snapshot
WHEN the snapshot is parsed
THEN MotorDto has no currentSpeedKmh field
AND MotorInfo has no currentSpeedKmh field
```

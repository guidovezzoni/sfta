# 2.1 Fix Repository Clean Architecture Violation

## User Story

**As a** developer maintaining the rider dashboard,
**I want** the domain layer to have no dependency on data-layer types,
**So that** the architecture follows Clean Architecture principles and the layers can evolve independently.

## Description

The `BikeInfoRepository` interface in the domain layer currently imports and returns `Result<BikeInfoSnapshotDto>` — a data-layer DTO. In Clean Architecture, the domain layer must not depend on the data layer. The interface should return `Result<BikeInfo>` (the domain model), with the DTO-to-domain mapping happening inside the repository implementation.

This also means the mapper, which converts data-layer types to domain types, belongs in the data layer rather than the domain layer.

## Detailed Scope

### 1. Repository Interface (`domain/repository/BikeInfoRepository.kt`)

**Current state:** imports `com.guidovezzoni.sfta.data.model.BikeInfoSnapshotDto` and returns `Result<BikeInfoSnapshotDto>`.

**Target state:**
- Change return type to `Result<BikeInfo>`
- Add import for `com.guidovezzoni.sfta.domain.model.BikeInfo`
- Remove import for `BikeInfoSnapshotDto`
- No other changes — the interface remains a single `suspend fun getBikeInfoSnapshot(): Result<BikeInfo>`

**Verification:** After the change, `BikeInfoRepository.kt` must contain zero imports from `com.guidovezzoni.sfta.data.*`.

### 2. Mapper Relocation (`domain/mapper/` → `data/mapper/`)

**Current state:** `BikeInfoMapper.kt` lives in `domain/mapper/` with package `com.guidovezzoni.sfta.domain.mapper`.

**Target state:**
- Use `git mv` to move `app/src/main/java/com/guidovezzoni/sfta/domain/mapper/BikeInfoMapper.kt` → `app/src/main/java/com/guidovezzoni/sfta/data/mapper/BikeInfoMapper.kt`
- Update the `package` declaration to `com.guidovezzoni.sfta.data.mapper`
- The extension function `BikeInfoSnapshotDto.toDomain()` and its mapping logic remain unchanged
- Delete the now-empty `domain/mapper/` directory if no other files remain

### 3. Repository Implementation (`data/repository/LocalBikeInfoRepository.kt`)

**Current state:** parses JSON into `BikeInfoSnapshotDto` and returns it directly.

**Target state:**
- Import `com.guidovezzoni.sfta.data.mapper.toDomain`
- After parsing the JSON into `BikeInfoSnapshotDto`, call `.toDomain()` to map to `BikeInfo`
- Return `Result<BikeInfo>` instead of `Result<BikeInfoSnapshotDto>`
- The `runCatching` block now covers both parsing and mapping — any mapping error will also be caught

### 4. Use Case Simplification (`domain/usecase/GetBikeInfoUseCase.kt`)

**Current state:** calls `repository.getBikeInfoSnapshot().map { it.toDomain() }` — the use case performs the DTO-to-domain mapping.

**Target state:**
- Remove the `.map { it.toDomain() }` call — the repository already returns `BikeInfo`
- Remove the import of `com.guidovezzoni.sfta.domain.mapper.toDomain`
- The use case becomes a pass-through: `repository.getBikeInfoSnapshot()`
- Keep the use case class to maintain the use case pattern for future orchestration logic

### 5. Test Updates

#### `data/repository/LocalBikeInfoRepositoryTest.kt`
- The success-path test must now assert domain model fields (`BikeInfo`) instead of DTO fields
- Rename assertion variable from `dto` to `bikeInfo` (or similar) for clarity
- Verify key domain-mapped fields: `bike.model`, `battery.stateOfChargePercent`, `motor.currentSpeedKmh`, `rideSettings.powerMap` (as `PowerMap.ENDURO` enum), `session.durationSeconds`
- Failure-path tests remain structurally unchanged

#### `domain/mapper/BikeInfoMapperTest.kt` → `data/mapper/BikeInfoMapperTest.kt`
- Use `git mv` to relocate the test file
- Update the `package` declaration to `com.guidovezzoni.sfta.data.mapper`
- The test class, assertions, and logic remain unchanged

#### `domain/usecase/GetBikeInfoUseCaseTest.kt`
- The mock `BikeInfoRepository` now returns `Result<BikeInfo>` directly (not `Result<BikeInfoSnapshotDto>`)
- Replace the `sampleDto` (DTO construction) with a `sampleBikeInfo` (domain model construction)
- Remove all data-layer DTO imports
- Success test: mock returns `Result.success(sampleBikeInfo)`, assert the use case returns the same object unchanged
- Failure test: unchanged in structure

### 6. Spec and Documentation Updates

#### `openspec/specs/bike-telemetry-data/spec.md`

- **Use case requirement:** update to reflect pass-through behaviour — no longer chains mapper
- **Mapper requirement:** note its location in the data layer (`data/mapper/`)

- Update the Use Case section to reflect pass-through behaviour

## Security, Performance, and Non-Functional Considerations

### Security
- **No impact.** This is a pure refactoring story — no new data inputs, no network calls, no user-facing changes. The bundled JSON asset is read-only and contains no sensitive data.

### Performance
- **Negligible impact.** The mapper is now invoked inside `runCatching` within the repository rather than in the use case. Both run on the same coroutine dispatcher (`Dispatchers.IO`). No additional allocations or processing steps are introduced.

### GDPR / Sensitive Information
- **No impact.** The telemetry data is a static mock snapshot bundled in the APK. No personal user data is collected, stored, or transmitted.

### Other Non-Functional Requirements
- **Maintainability:** Improved. The domain layer becomes a pure business-logic layer with no data-layer coupling, making it easier to swap data sources in the future.
- **Testability:** Improved. Use case tests no longer need to construct DTOs — they work with domain models directly, making tests simpler and more readable.
- **Backward compatibility:** The `GetBikeInfoUseCase` public API (`suspend operator fun invoke(): Result<BikeInfo>`) is unchanged. Story 1.2's ViewModel and any future consumers are unaffected.

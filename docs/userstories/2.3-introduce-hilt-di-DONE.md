# 2.3 Introduce Hilt Dependency Injection

## User Story

**As a** developer scaling the rider dashboard app,
**I want** Hilt managing the dependency graph,
**So that** new features can declare their dependencies declaratively without manual wiring.

## Description

Manual DI works at the current project size but won't scale. The Android guidelines recommend introducing Hilt when the project grows. With stories 2.1 and 2.2 complete (data and domain layers are stable), this is the right time to introduce Hilt before building the Dashboard UI (story 1.2).

**Dependency:** Stories 2.1 and 2.2 must be complete (both are DONE).


### 2. Application Class

#### Create `app/src/main/java/com/guidovezzoni/sfta/SftaApplication.kt`

```kotlin
@HiltAndroidApp
class SftaApplication : Application()
```

#### Update `app/src/main/AndroidManifest.xml`

Add `android:name=".SftaApplication"` to the `<application>` element.

### 3. DI Module

#### Create `app/src/main/java/com/guidovezzoni/sfta/di/AppModule.kt`

A Hilt `@Module` annotated with `@InstallIn(SingletonComponent::class)` providing:

| Binding | Type | Scope | Implementation |
|---------|------|-------|----------------|
| `AssetManager` | `@Provides` | Unscoped | From `@ApplicationContext context: Context` |
| `BikeInfoRepository` | `@Provides` | `@Singleton` | Returns `LocalBikeInfoRepository(assetManager)` |

**Design decision — `@Provides` vs `@Inject constructor` for `LocalBikeInfoRepository`:**
`LocalBikeInfoRepository` takes an `ioDispatcher: CoroutineDispatcher = Dispatchers.IO` parameter. Using `@Inject constructor` would require Hilt to resolve all constructor parameters, forcing a `@Qualifier` annotation for the dispatcher. Using `@Provides` in the module lets us construct the repository with its default dispatcher, keeping the DI setup minimal. This is the right trade-off at the current project size.

`GetBikeInfoUseCase` does **not** need a module entry — its `@Inject constructor` allows Hilt to construct it automatically by resolving its `BikeInfoRepository` dependency from the module binding.

### 4. Existing Class Modifications

#### `domain/usecase/GetBikeInfoUseCase.kt`

Add `@Inject constructor`:

```kotlin
class GetBikeInfoUseCase @Inject constructor(
    private val repository: BikeInfoRepository,
)
```

This is backward-compatible — manual construction in existing tests continues to work.

**Note:** Adding `javax.inject.Inject` to the domain layer introduces a dependency on JSR-330, which is a standard Java annotation spec (not Hilt-specific). This is widely accepted in Android clean architecture.

#### `data/repository/LocalBikeInfoRepository.kt`

**No annotation changes.** The module constructs it via `@Provides`, so `@Inject constructor` is not needed.

### 5. Activity

#### `ui/MainActivity.kt`

- Add `@AndroidEntryPoint` annotation.
- The activity currently has no manual dependency construction, so no removal is needed. The annotation prepares it for ViewModel injection in story 1.2.

### 6. Spec and Documentation Updates

#### `openspec/specs/bike-telemetry-data/spec.md`

Add a new requirement section documenting that the Repository and Use Case are provided via Hilt dependency injection.

#### `docs/userstories/1.1-data-infrastructure-DONE.md`

Update relevant sections to note that dependencies are now Hilt-injected.

## Security Considerations

**No impact.** Hilt is a compile-time DI framework. It does not expose new attack surfaces, network endpoints, or data storage. All dependency wiring is resolved at compile time via annotation processing.

## Performance Considerations

- **Runtime:** Minimal impact. Hilt generates code at compile time — no runtime reflection. App startup shows a negligible increase due to Hilt component initialisation, offset by `@Singleton` scoping of the repository.
- **Build time:** KSP annotation processing adds a small overhead to the build. Acceptable for the benefits of automated DI.

## GDPR and Sensitive Information

**No impact.** This is a purely structural change. No new data is collected, stored, or transmitted. No user-facing behaviour changes.

## Other Non-Functional Requirements

- **Maintainability:** Hilt reduces boilerplate and makes the dependency graph explicit, improving maintainability as the project grows.
- **Testability:** Hilt supports test-specific modules (`hilt-android-testing`) for easy dependency substitution in integration tests. This can be added in future stories.
- **Compatibility:** KSP version must match the project's Kotlin version. If Kotlin is upgraded, KSP must be upgraded accordingly.

# 1.2 Data Model Null Safety

## User Story

**As a** developer integrating real-time telemetry from the Stark VARG,
**I want** every data model field to handle null values gracefully,
**So that** the app never crashes when the bike reports incomplete telemetry during real data collection.

## Background

The JSON snapshot (`app/src/main/assets/bike_info_snapshot.json`) no longer provides `current_speed_kmh` in the motor object. However, `MotorDto.kt` still declares a non-nullable `currentSpeedKmh: Double` field with no default value. Since `kotlinx-serialization` throws `MissingFieldException` when a required field is absent from JSON, **the current app crashes at runtime** when parsing the motor object. This field must be removed from the DTO, domain model, mapper, and all test fixtures.

Additionally, when real data collection is implemented, any JSON field could be `null` (sensor offline, data not yet available, partial telemetry). The data layer must deserialise these without crashing, and the domain layer must surface nullability so the UI can decide how to present missing values.

## Detailed Scope

### Phase 1: Remove `currentSpeedKmh`

Remove the `currentSpeedKmh` field entirely from the data and domain layers:

| File | Change |
|------|--------|
| `app/src/main/java/com/guidovezzoni/sfta/data/model/MotorDto.kt` | Remove the `@SerialName("current_speed_kmh") val currentSpeedKmh: Double` field |
| `app/src/main/java/com/guidovezzoni/sfta/domain/model/MotorInfo.kt` | Remove the `val currentSpeedKmh: Double` field |
| `app/src/main/java/com/guidovezzoni/sfta/data/mapper/BikeInfoMapper.kt` | Remove `currentSpeedKmh = motor.currentSpeedKmh` mapping line |

### Phase 2: Make DTO fields nullable

All DTO fields become nullable with `= null` defaults. This tells `kotlinx-serialization` the field is optional â€” missing or `null` in JSON both produce `null` in the DTO.

| File | Fields to make nullable |
|------|------------------------|
| `data/model/BikeInfoSnapshotDto.kt` | `bike: BikeDto? = null`, `timestamp: String? = null`, `battery: BatteryDto? = null`, `motor: MotorDto? = null`, `rideSettings: RideSettingsDto? = null`, `session: SessionDto? = null`, `diagnostics: DiagnosticsDto? = null` |
| `data/model/BikeDto.kt` | `model: String? = null`, `variant: String? = null`, `firmwareVersion: String? = null`, `imageUrl: String? = null` |
| `data/model/BatteryDto.kt` | `stateOfChargePct: Int? = null`, `estimatedRangeKm: Int? = null`, `temperatureC: Double? = null`, `chargingState: String? = null` |
| `data/model/MotorDto.kt` | `powerHp: Double? = null`, `temperatureC: Double? = null` |
| `data/model/RideSettingsDto.kt` | `powerMap: String? = null`, `maxPowerHp: Int? = null`, `engineBrakingPct: Int? = null`, `regenPct: Int? = null` |
| `data/model/SessionDto.kt` | `durationS: Long? = null`, `distanceKm: Double? = null`, `maxSpeedKmh: Double? = null` |
| `data/model/DiagnosticsDto.kt` | `faultCodes: List<String>? = null`, `warnings: List<WarningDto>? = null` |
| `data/model/WarningDto.kt` | `code: String? = null`, `message: String? = null`, `severity: String? = null` |

### Phase 3: Make domain model fields nullable

Domain models mirror the DTO nullability so the UI layer can decide how to render missing data (e.g. "--", hide a panel, show a placeholder).

| File | Fields to make nullable |
|------|------------------------|
| `domain/model/BikeInfo.kt` | `bike: BikeDetails?`, `timestamp: Instant?`, `battery: BatteryInfo?`, `motor: MotorInfo?`, `rideSettings: RideSettingsInfo?`, `session: SessionInfo?`, `diagnostics: DiagnosticsInfo?` |
| `domain/model/BikeDetails.kt` | `model: String?`, `variant: String?`, `firmwareVersion: String?`, `imageUrl: String?` |
| `domain/model/BatteryInfo.kt` | `stateOfChargePercent: Int?`, `estimatedRangeKm: Int?`, `temperatureCelsius: Double?`, `chargingState: ChargingState?` |
| `domain/model/MotorInfo.kt` | `powerHp: Double?`, `temperatureCelsius: Double?` |
| `domain/model/RideSettingsInfo.kt` | `powerMap: PowerMap?`, `maxPowerHp: Int?`, `engineBrakingPercent: Int?`, `regenPercent: Int?` |
| `domain/model/SessionInfo.kt` | `durationSeconds: Long?`, `distanceKm: Double?`, `maxSpeedKmh: Double?` |
| `domain/model/DiagnosticsInfo.kt` | `faultCodes: List<String>?`, `warnings: List<WarningInfo>?` |
| `domain/model/WarningInfo.kt` | `code: String?`, `message: String?`, `severity: WarningSeverity?` |

**Enum nullability semantics:**
- `null` = no data received from sensor (field absent or explicitly `null` in JSON)
- `UNKNOWN` = data received but value not recognised (non-null string that doesn't match any enum entry)

### Phase 4: Update mapper

Update `data/mapper/BikeInfoMapper.kt` â€” the `toDomain()` extension function:

| Scenario | Mapping rule |
|----------|-------------|
| Null sub-object in DTO (e.g. `battery == null`) | `null` in domain (`battery = dto.battery?.let { mapBattery(it) }`) |
| Null primitive field | `null` in domain (direct assignment) |
| Null enum string (e.g. `chargingState == null`) | `null` in domain â€” **NOT** `UNKNOWN` |
| Non-null unrecognised enum string | `UNKNOWN` (existing behaviour preserved) |
| Null timestamp string | `null` `Instant` |
| Null list fields (e.g. `faultCodes == null`) | `null` in domain â€” **NOT** empty list |

The mapper should use safe-call chains and `?.let {}` to handle nullable sub-objects.

## Non-Functional Considerations

### Security
No security concerns. This story modifies data model types and a local JSON parser. No network calls, no user input, no authentication, no data storage. The null-safety improvements actually reduce crash surface area.

### Performance
No performance impact. Nullable types in Kotlin compile to the same bytecode (boxed types for primitives, same reference types for objects). The `?.let {}` calls in the mapper add negligible overhead. No new allocations or loops.

### GDPR / Sensitive Information
No GDPR concerns. The data models represent bike telemetry (battery, motor, ride settings) â€” no personally identifiable information. The JSON snapshot is a bundled asset, not user-generated data.

### Forward Compatibility
This change is the primary forward-compatibility measure: by making all fields nullable with defaults, the app will gracefully handle any future JSON schema changes (new fields, removed fields, null values) without crashing. The `ignoreUnknownKeys = true` already configured in `LocalBikeInfoRepository` handles new fields; nullable defaults handle missing/null fields.
